# Техническое задание и план работ

**По итогам созвона от 10 июля 2026 года**
**AI-платформа массового управления Telegram-профилями**

## 1. Цель проекта

Создать масштабируемую платформу управления большим количеством Telegram-профилей
с безопасной сетевой инфраструктурой, прогревом, нейрокомментингом, нейрочатингом,
парсингом каналов, автоматизациями, централизованными логами и аналитикой.
Архитектура должна допускать дальнейшее подключение **Instagram** и генеративного
контура на базе **ComfyUI**.

## 3. Полный список задач

### 3.1. Логи, история и Help Center
- Унифицировать терминологию: заменить «История» на «Логи» во всех модулях.
- Интегрировать историю действий и комментариев в единый экран логов; не плодить дубли.
- Заменить карточки логов на раскрывающийся список с multi-select (2–3 потока параллельно).
- Добавить детальную страницу/вью записи истории и комментариев с переходом из списка.
- Расширить состав логов: по массовой ошибке видно действие, модуль, профиль, инициатор, время, причина/код и масштаб затронутых профилей.
- Ссылки «?» открывают Help Center сразу на нужном якоре.
- Help Center — правая боковая панель, а не блокирующая модалка; основной интерфейс доступен.

### 3.2. Менеджер профилей и права
- Переименовать «Управление аккаунтами» → «Менеджер профилей» (профиль — человек/пользователь системы, аккаунт — управляемый бот).
- На экране модуля показывать только свободные/разрешённые профили; чужие не захватывать незаметно.
- При переносе профилей между модулями — подтверждение с последствиями; фиксировать инициатора и действие в аудите.
- Для опасных массовых действий — двухэтапное подтверждение; второе окно визуально смещать; «не спрашивать повторно» только по согласованным ролям.
- Фильтры: страна, роль/модуль, статус, прогрев, исправность прокси.
- Страны в фильтре — из реально подключённых номеров/профилей.
- Состояние прокси и его валидацию вынести в Менеджер профилей; сохранить фильтр «рабочий прокси».
- Позже — роли пользователей и ограничения видимости папок, каналов, экспортов, админ-операций.

### 3.3. Прогрев, защита и карантин
- Профиль в статусе «Прогрев» полностью заблокирован для ручного назначения и других модулей; действия определяет сценарий прогрева.
- Прогрев смешивает естественные действия из разных модулей: поиск, чтение, комментарии, чаты.
- При падении рейтинга доверия/подозрительной активности — автоматически остановить рабочие задачи и вернуть в прогрев.
- После восстановления рейтинга — авто-возврат (или после проверки) в доступный пул.
- **Исключение:** активный горячий лид — диалог продолжается даже в прогреве; статус не менять.
- Развести статусы **Pause / FloodWait / Quarantine / SpamBlock**; хранить причину, длительность, разрешённые действия.
- Число попыток до карантина — настраиваемое (обсуждалось 3); длительность ограничения брать от Telegram.
- Подготовить ≥ 5 тестовых профилей для проверки блокировок, FloodWait, карантина и горячего лида.

### 3.4. Прокси, SIM и регистрация
- Все аккаунты — через назначенный прокси; прямое подключение — только временное тестовое.
- Описать модель: аккаунт → страна/город → SIM → proxy endpoint → device/IMEI → статус.
- Сравнить статические и мобильные прокси (ротация 3–5 мин); оценить собственную мобильную proxy-ферму.
- Найти DevOps или подтвердить, что инфраструктуру соберёт команда.
- Исследовать поставщиков физических SIM в UA/EU; цены для 100/500/1000+ карт и правила продления.
- Виртуальные номера — не как основной долгоживущий контур (слабый контроль, риск возврата).
- Спроектировать масштабируемую регистрацию на физических Android/ферме; получение кодов; восстановление разлогиненных.
- Ограниченный пилот до массовой закупки; измерить срок жизни, % блокировок, стоимость регистрации и сопровождения.

### 3.5. Нейрокомментинг — интерфейс и логика
- Явно показывать Conservative/Balanced/Aggressive и фактическое влияние на задержки и защиту.
- На финальном экране — режим и расчётное min/avg/max время вместо абстрактного «интервала».
- Связать общий максимум комментариев, число аккаунтов и min/max на аккаунт; предупреждать о противоречиях.
- Ограничить обработку постов окном (последние 3/5/10 или настраиваемое); не обходить всю историю канала.
- Фильтрация постов/комментариев по тональности/негативу; согласовать режим семантического поиска (простой фильтр / вектор / оба).
- Распределение типов комментариев в процентах; сумма категорий = 100%.
- Глобальный системный промпт для запретов/правил; отдельные промпты — для тона/направления.

### 3.6. Цели, база знаний, CRM и нейрочатинг
- Отдельная страница «Цели»: название, описание целевого действия, этапы, критерий завершения, аудитория.
- К цели — база знаний: текст, файлы, изображения; версионирование и область использования.
- В AI-промптах выбирать цель из списка; все боты кампании двигаются к одной цели.
- Разделить: системные ограничения, база знаний о продукте, цель кампании, стиль сообщения.
- CRM-логика: лид, статус (холодный / ответил / горячий / целевое действие / закрыт), ответственный аккаунт, история диалога, результат.
- Лимиты нейрочатинга учитывают число **активных диалогов**, а не только общий максимум сообщений.
- После ответа лида — приоритет диалогу; аккаунт продолжает беседу до цели с отдельными safety-лимитами.
- Не обрывать активную беседу из-за общего лимита кампании; не спамить неответившим.
- Аналитика: отправлено, ответы, активные диалоги, горячие лиды, целевые действия, конверсия по целям/аккаунтам.

### 3.7. Выбор целевых каналов
- После выбора папок — объединённый список каналов, а не назначение всей папки вслепую.
- Для канала: название, ссылка/ID, категория, язык, регион, подписчики, активность, доступность комментариев, рейтинг.
- Поиск и фильтры: название, страна/регион, язык, тематика, размер аудитории, активность, комментарии.
- Разделять язык общения и регион канала (не выводить друг из друга).
- Multi-select каналов + массовое добавление/исключение; учитывать будущие права видимости.
- Упрощённую версию списка сделать общей для нейрокомментинга, нейрочатинга и др.

### 3.8. Парсер каналов
- «Шаблоны» → «Шаблоны поиска».
- Создание/сохранение пользовательских шаблонов; позже — доступ по ролям.
- Показывать активные шаблоны и состояние «применён».
- Несколько шаблонов — **пересечение/AND** (IT + Crypto), а не сумма выдач.
- «Быстрая работа» перенести внутрь блока задержек; при включении скрывать/блокировать поля и объяснять эффект.
- Каждый найденный канал — в общей базе; дубли не создавать, а привязывать к доп. категориям/запускам.
- Переработать блок фильтров; визуально разделить настройки, фильтры, статистику запуска и результаты.
- В результатах разделить действия: (фильтр/поиск/очистка/копирование) и (папка/Excel/JSON/экспорт).
- Для админов — все форматы экспорта; право экспорта позже по ролям.
- Сохранять язык, регион, подписчиков, рейтинг, наличие комментариев, активность постов/комментариев.
- «Каналы без комментариев» — не единственный сценарий; наличие комментариев — статистический признак.
- Сохранить фильтры диапазона участников и рейтинга; определить собственную прозрачную формулу рейтинга.
- «Открыть в Telegram» + корректное копирование ID/ссылки.

### 3.9. Задачи, автоматизации и статистика каналов
- Каждый запуск модуля создаёт задачу: ID, модуль, цель, профили, каналы, время, статус, прогресс.
- Внешний трекер задач: просмотр, пауза, продолжение, отмена, перезапуск, лог ошибок, результат.
- Один профиль не выполняет конфликтующие задачи; статус прогрева — приоритет.
- Автоматизация по расписанию включает/выключает модули и перераспределяет группы профилей в рамках цели.
- Модуль обновления статистики каналов: выбор аккаунтов, каналов, задержек.
- Уникальные пары «один канал — один доступный бот»; несколько ботов не обновляют один канал.
- Первый проход — боты, уже работающие в каналах; второй — назначать доступные на необслуженные.
- Блокировка/lease задачи против гонки в миллисекундах.
- Регулярное обновление (сутки / 3 дня / неделя — **открытый вопрос**) + ручная кнопка «обновить сейчас».
- Статистика активности постов/комментариев; сохранять дату последнего обновления и источник.

### 3.10. Дальнейшее развитие
- Платформенный переключатель Telegram/Instagram в архитектуре парсеров и кампаний.
- Не переносить Telegram-специфичные поля в общий слой без адаптера платформы.
- Исследовать ComfyUI (comfy.org) для генерации контента; определить место в пайплайне.
- К генеративному контуру вернуться после стабилизации Telegram-модулей, целей и базы знаний.

## 4. Ключевые бизнес-правила

| Правило | Требование |
|---|---|
| Прогрев выше рабочих задач | Непрогретый или повторно отправленный на прогрев профиль недоступен всем рабочим модулям. |
| Горячий лид — исключение | Активный горячий диалог продолжается даже при прогреве, но с ограничениями защиты. |
| Перенос только явно | Переключение аккаунта между модулями требует подтверждения и записи в аудит. |
| Один исполнитель на единицу работы | Один канал в конкретный момент обновляет только один бот; нужен lock/lease. |
| AI работает к цели | Промпт без цели и продуктовой базы знаний не считается готовой кампанией. |
| Активный диалог приоритетнее массового лимита | Ответивший лид получает ресурс для продолжения беседы; неответившие не спамятся. |
| Данные канала не теряются | Повторный парсинг обновляет карточку и связи, а не создаёт дубль. |
| Язык не равен региону | Оба параметра определяются и хранятся отдельно. |

## 5. Критерии готовности ближайшего релиза

1. Нельзя назначить непрогретый профиль или незаметно забрать профиль из другого модуля.
2. Каждое массовое действие и перенос профилей — в логах с инициатором и результатом.
3. Help Center открывается справа и переходит к нужной статье без блокировки интерфейса.
4. Из нескольких папок можно выбрать конкретные каналы через список и фильтры.
5. Парсер не создаёт дубли, сохраняет язык/регион/метрики и объединяет шаблоны через AND.
6. Каждый запуск — управляемая задача со статусом и прогрессом.
7. AI-кампания привязана к цели, правилам и базе знаний; активный лид не теряется из-за лимита.
8. Модуль статистики гарантирует единственного исполнителя на канал и показывает дату актуальности.

## 7. Рекомендуемая декомпозиция первого спринта (из ТЗ)

- **День 1–2:** state machine аккаунта, схема профилей/ролей/прокси, правила блокировок.
- **День 2–4:** выбор профилей, подтверждения переноса, аудит, терминология.
- **День 3–5:** логи и Help Center; детальная вью.
- **День 4–7:** список целевых каналов из нескольких папок и основные фильтры.
- **День 6–9:** шаблоны поиска, задержки и результаты парсера.
- **День 8–10:** базовая сущность Task и трекер запуска.
- **Параллельно:** архитектурный документ Goals/Knowledge Base/Leads + сравнительный отчёт SIM/proxy.

> Сроки — ориентировочная последовательность, а не оценка трудозатрат. Оценку в часах
> дать после аудита текущего кода и схемы данных (см. ROADMAP → Фаза 0).

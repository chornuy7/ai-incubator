# Контракт: единый лог и аудит

> **Фаза 0 · Lane A+B.** Требования: [TZ-2026-07-10.md](./TZ-2026-07-10.md) §3.1, §5, §4.
> Расширяет запись лога из [`server/lib/taskStore.js`](../server/lib/taskStore.js) (`appendLog`) —
> обратносовместимо. Терминология: **«Логи»**, не «История» (§3.1, CLAUDE.md).

## 1. Зачем

§3.1/§5.2: по массовой ошибке и по любому переносу профиля должно быть видно **действие, модуль,
профиль, инициатор, время, причину/код и масштаб затронутых профилей**. Сейчас лог-запись беднее.

## 2. Текущая запись (что уже есть)

`appendLog(task, level, message, account?)` пишет в `task.logs`:
```jsonc
{ "id", "ts": "ISO", "level": "info|success|warning|error", "message", "account?" }
```
Хранится в задаче (`data/modules/<module>/tasks/<id>.json`), кап 500 записей. **Это сохраняем.**

## 3. Целевая запись (расширение)

Добавляем поля **поверх** существующих (старые записи без них валидны):

```jsonc
{
  // — существующие —
  "id": "8f2c1a0b",
  "ts": "2026-07-13T10:20:30.000Z",
  "level": "info|success|warning|error",
  "message": "человекочитаемое описание",
  "account": "<accountId>?",           // конкретный бот (если применимо)

  // — новые (§3.1/§5) —
  "action": "comment.sent | task.start | task.stop | account.status.change | account.transfer | lock.force | lease.acquire | proxy.check | parse.result | ...",
  "module": "<moduleKey>",             // neuro-commenting | parsing | warming | ...
  "initiator": "<profileId> | system | scheduler",  // КТО инициировал (§4 аудит)
  "code": "FLOOD_WAIT_420 | USER_BANNED | ACCOUNT_BUSY | OK | ...",  // машинный код/причина
  "scope": {                            // масштаб затронутого (§3.1 «масштаб затронутых профилей»)
    "accounts": ["<id>", ...],          // сколько/какие аккаунты затронуты
    "channels": ["<id>", ...],
    "taskId": "<id>"
  },
  "reason": "текст причины?"            // доп. пояснение к code
}
```

Минимально обязательные для **аудит-событий** (перенос/статус/массовое действие): `action`, `module`,
`initiator`, `ts`, `code`/`reason`, `scope`. Для обычных операционных логов достаточно старого набора.

## 4. Классы событий (`action`)

| Префикс | Примеры | Уровень аудита? |
|---|---|---|
| `task.*` | `task.start`, `task.pause`, `task.stop`, `task.done`, `task.error` | да (initiator) |
| `account.status.*` | `account.status.change` (active→floodwait…) | **да** (§4, обяз. initiator+reason) |
| `account.transfer` | перенос профиля между модулями | **да** (§3.2/§4 — подтверждение + инициатор) |
| `account.mass` | двухэтапно подтверждённое массовое действие | **да** (§3.2) |
| `lock.*` / `lease.*` | `lock.acquire`, `lock.force`, `lease.acquire`, `lease.expire` | да (для force/transfer) |
| `proxy.*` | `proxy.check`, `proxy.fail` | нет |
| `comment.* / chat.* / react.* / look.*` | операционные действия воркеров | нет |
| `parse.*` | `parse.result`, `parse.dedup` | нет |

## 5. Два уровня чтения (§3.1)

1. **Операционные логи задачи** — как сейчас, внутри `task.logs`, показываются на экране модуля.
2. **Единый экран логов** (§3.1, Lane B) — агрегирует `logs` всех задач всех модулей в один список
   с multi-select (2–3 потока), раскрывающийся список + детальная вью записи. Источник — те же `task.logs`,
   агрегатор уже частично есть (`ParsingHistoryPage`/`fetchParsingHistory`). **Новых хранилищ не вводим** —
   только обогащаем запись и добавляем сборку/фильтр.

**Аудит-события** (класс «да» выше) дополнительно можно дублировать в отдельный append-only файл
`data/audit.log.jsonl` (для неизменяемого следа переносов/статусов) — 🔒 решение о необходимости
отдельного файла оставляем на Lane A (по умолчанию: хватает `task.logs` + фильтр по `action`).

## 6. Инварианты (§5.2)

1. Любое `account.transfer`, `account.status.change` (в нерабочий статус), `account.mass`, `lock.force`
   пишет запись с **непустым `initiator`** и `reason`/`code`.
2. `scope.accounts` отражает реальный масштаб (для массовой ошибки — все затронутые id).
3. Формат `ts` — ISO-8601 UTC (как сейчас). Сортировка/фильтр — по `ts`, `module`, `action`, `initiator`.
4. Обратная совместимость: читатель логов не падает на записях без новых полей (трактует как `initiator:"system"`, `action:"legacy"`).

# Архитектура: Цели → База знаний → Лиды/CRM

> **Фаза 0 · Lane A+B (владение — Lane B).** Требования: [TZ-2026-07-10.md](./TZ-2026-07-10.md) §3.6 (P0), §4.
> Data flow слоя целей: как нейрочатинг ведёт диалог **к цели**, а не генерит случайные сообщения (§4).
> Схема сущностей — в [SCHEMA-entities.md](./SCHEMA-entities.md). Это контракт потоков, не реализация.

## 1. Идея (§4 «AI работает к цели»)

Промпт **без цели и продуктовой базы знаний не считается готовой кампанией**. Три разделённых слоя (§3.6):

```
Ограничения (что нельзя)  +  База знаний (что знаем о продукте)  +  Цель (куда ведём)  +  Стиль (как говорим)
        системный промпт            KnowledgeBase                     Goal.stages          отдельный промпт
```
Раньше эти четыре вещи смешивались в одном промпте — контракт требует их **развести** (§3.6).

## 2. Сущности слоя (кратко; поля — в SCHEMA-entities.md)

- **Goal** — название, целевое действие, этапы, критерий завершения, аудитория.
- **KnowledgeBase** — привязанные к цели текст/файлы/изображения; версионирование; область использования.
- **Lead** — лид: статус, ответственный аккаунт, история диалога, результат.

## 3. Data flow кампании

```
      ┌── Оператор создаёт Goal ──┐         ┌── прикрепляет KnowledgeBase (версии) ──┐
      ▼                           ▼         ▼                                        │
   [ Страница «Цели» ] ── goalId ──> [ Кампания нейрочатинга ] <── выбор Goal из списка (§3.6)
                                            │
                                            │  все боты кампании двигаются к ОДНОЙ цели (§3.6)
                                            ▼
                    ┌────────── для каждого целевого peer ──────────┐
                    ▼                                                │
             создаётся/находится Lead(status=cold)                  │
                    │                                               │
        бот шлёт сообщение (в рамках лимитов активных диалогов, §3.6)│
                    │                                               │
        peer ответил? ──нет──> не спамим (§4 «неответившие не спамятся»)
                    │ да
                    ▼
             Lead.status = answered → hot   (приоритет диалогу, §3.6)
                    │
        бот ведёт беседу к Goal.completionCriteria, используя KnowledgeBase
        с отдельными safety-лимитами (🔒 §6), НЕ обрывая из-за общего лимита (§4)
                    │
                    ▼
             target (целевое действие) ──> closed
                    │
                    ▼
             Аналитика: отправлено / ответы / активные диалоги / горячие лиды / конверсия (§3.6)
```

## 4. Связь с ядром (Lane A) — точки контракта

- **Lead.isHot → State machine (§3.3):** активный горячий лид — исключение: аккаунт продолжает диалог
  даже в статусе `warming`, статус не меняем (см. [CONTRACT-account-state-machine.md](./CONTRACT-account-state-machine.md) §6).
  Механизм «как технически отвечать при прогреве» — 🔒 §6.
- **Лимиты по активным диалогам (§3.6):** нейрочатинг считает **число активных Lead(status∈{answered,hot})**,
  а не только общий максимум сообщений. Приоритет ресурса — ответившим.
- **Task.goalId (§3.9):** запуск кампании = Task, привязанный к `goalId` (см. SCHEMA-entities Task).
- **Аудит (§4):** смена `Lead.status` и назначение ответственного аккаунта → запись в единый лог
  (`action: lead.status`, см. [CONTRACT-audit-log.md](./CONTRACT-audit-log.md)).

## 5. Что уже есть / чего нет

- **Есть:** нейрочатинг/НейроДиалоги-воркеры (`sendMessage`, контекст диалога), генерация OpenAI.
  Это переиспользуем как «исполнителя».
- **Нет:** сущностей Goal/KnowledgeBase/Lead, страницы «Цели», привязки промпта к цели, CRM-статусов,
  учёта активных диалогов, аналитики по целям. Скелет бэкенда + страница «Цели» — **Фаза 1** (§3.6, P0);
  CRM/аналитика — **Фаза 3**.

## 6. Границы Фазы 0

Фаза 0 фиксирует **имена, статусы Lead, направление потока и точки стыка с ядром**. Не реализуем:
хранилище, UI, семантику подбора. Дальше Lane B делает бэкенд-скелет по этому контракту.
